## Introduction

This post documents a basic template for a Next.js project that I am personally using, heavily referencing the video by CJ from Syntax: [Create your own Next.js Starter Template](https://youtu.be/dLRKV-bajS4?si=MAeBfZa6-uoo1wa_). At the time of writing, I have already completed another [project which involved rolling my own authentication from scratch in Next.js](../auth-from-scratch), but I still haven't decided whether to continue using this custom auth for other Next.js projects, or simply use a third-party service like Clerk or Better-Auth. I plan to try my custom auth on a simpler, smaller Next.js project first before deciding on its suitability for future projects.

## Setup

The `eslint-config-prettier` package helps ESLint and Prettier work together by disabling any ESLint rules that might conflict with Prettier.

Create a `.prettierrc.json` file, modify the existing `eslint.config.mjs`, and create a hidden `.vscode` folder containing a `settings.json` file. This setup enables automatic code formatting and import sorting in VS Code on save.

Here are the configurations for Prettier, ESLint, and VS Code settings:

## Dark Mode

Dark mode is a common feature in modern projects. I use the standard `shadcn/ui` solution, following the instructions from the [shadcn/ui documentation](https://ui.shadcn.com/docs/dark-mode). The main difference is that I created a `provider.tsx` file where I set all the necessary attributes for `next-themes`, instead of setting them directly in the main `layout.tsx`. This approach is useful for projects requiring multiple providers, as they can all be consolidated within this single `provider.tsx` file, keeping the main `layout.tsx` minimal. Remember to still set the `suppressHydrationWarning` attribute on the `html` tag in the main `layout.tsx`.

Another difference is the theme toggle component. Instead of a dropdown or a standard button, I use the `Toggle` component from `shadcn/ui`. It only switches between dark and light modes (no system mode option) with a single click. This is my personal preference, and you can customize it as needed. I also slightly modified the default `Toggle` style to remove the background color that indicates its `data-[state=on]` status.

<ScriptCopy
  scripts={[
    {
      packageName: "provider.tsx",
      script:
`"use client";

import { ThemeProvider } from "next-themes";

export function Providers({ children }: { children: React.ReactNode }) {
    return (
      <ThemeProvider
        attribute="class"
        enableSystem
        defaultTheme="system"
        disableTransitionOnChange
      >
        {children}
      </ThemeProvider>
    );
}
`
    },
    {
      packageName: "theme-toggle.tsx",
      script:`"use client";

import { Moon, Sun } from "lucide-react";
import { useTheme } from "next-themes";

import { Toggle } from "./ui/toggle";

export const ThemeToggle = () => {
  const { theme, setTheme } = useTheme();
  return (
    <Toggle onClick={() => setTheme(theme === "dark" ? "light" : "dark")}>
      <Sun className="h-[1.2rem] w-[1.2rem] scale-100 rotate-0 transition-all dark:scale-0 dark:-rotate-90" />
      <Moon className="absolute h-[1.2rem] w-[1.2rem] scale-0 rotate-90 transition-all dark:scale-100 dark:rotate-0" />
      <span className="sr-only">Toggle theme</span>
    </Toggle>
  );
};
`
    },
    {
      packageName: "toggle.tsx",
      script:
`//remove this from the toggleVariant className
data-[state=on]:bg-accent data-[state=on]:text-accent-foreground`
    },
  ]}
/>

## Environment Variable

While the standard way of declaring and using environment variables works, certain libraries or packages can sometimes have issues recognizing variable types, forcing workarounds like appending a `!` assertion in TypeScript. Typographical errors in variable names are also a common pitfall.

To address this, I install `@t3-oss/env-nextjs` and `zod`. This setup provides type-safe environment variables, significantly reducing errors and offering helpful IntelliSense auto-completion when accessing variables. I haven't customized the setup significantly; I split the configuration into `server.ts` and `client.ts` files but currently only use the `server.ts` file. For configuration details, you can follow the official [T3 Env documentation](https://env.t3.gg/docs/nextjs).

## Docker Database

Many online hosted databases exist, like MongoDB, Turso, Neon, and others. However, I've previously used services that offered free tiers initially but later discontinued them. Consequently, I now prefer using Docker to run a local PostgreSQL instance for development. That said, at the time of writing, I do plan to use Neon for some projects intended for deployment on Vercel once they are completed.

Here is my simple Docker setup for reference:
